# old 代码手动升级到新代码指南

本文档的目标是：帮助你把 `old/` 目录中的旧代码，**手动**改成当前根目录 `src/`、`include/` 中的目标新代码。

本文档不是简单贴一个大 diff，而是按“功能模块 + 修改顺序”拆开讲。

你可以把它当成一份施工手册。

---

## 一、你现在手里的两套代码

### 1.1 旧代码位置

```text
old/include/
old/src/
```

### 1.2 目标新代码位置

```text
include/
src/
```

### 1.3 修改原则

建议你 **不要一开始就对着所有 diff 硬改**，而是按下面顺序改：

1. 先新增协议层文件
2. 再改头文件
3. 再改客户端入口和客户端命令处理
4. 再改服务端入口、线程池
5. 最后重写服务端 `handle.c`
6. 最后再同步注释型变化

这样做的原因很简单：

1. 新版代码的核心变化，是“协议重构”
2. `client_command_handle.c` 和 `server/handle.c` 都依赖新协议
3. `server.c / thread_pool.c / worker.c` 又依赖新的线程池结构

如果顺序乱了，你会改得很痛苦。

## 三、推荐修改顺序

### 第 1 步：新增协议层文件

要新增两个文件：

1. `include/protocol.h`
2. `src/common/protocol.c`

这两个文件是新版代码最核心的变化。

旧代码里，客户端和服务端很多地方都是：

1. 先发一个 `int len`
2. 再发命令字符串
3. 再单独发 `off_t`
4. 再单独发文本响应

这种协议很散，容易错位。

新代码的思路是：

1. 普通命令统一用 `command_packet_t`
2. 文件上传下载统一用 `file_packet_t`
3. 所有固定长度结构体都用 `send_full / recv_full` 完整收发

### 3.1 `include/protocol.h` 在旧代码中是全新文件

旧代码中：**没有这个文件**

新代码中：**新增这个文件**

你可以直接新建 `old/include/protocol.h`，然后手抄或复制目标文件。

核心结构如下：

```c
typedef enum {
    CMD_TYPE_INVALID = 0,
    CMD_TYPE_PWD,
    CMD_TYPE_CD,
    CMD_TYPE_LS,
    CMD_TYPE_GETS,
    CMD_TYPE_PUTS,
    CMD_TYPE_RM,
    CMD_TYPE_MKDIR,
    CMD_TYPE_REPLY
} cmd_type_t;

typedef struct {
    int cmd_type;
    int data_len;
    char data[CMD_DATA_LEN];
} command_packet_t;

typedef struct {
    int cmd_type;
    int data_len;
    off_t file_size;
    off_t offset;
    char file_name[FILE_NAME_LEN];
} file_packet_t;
```

你需要连同下面这些函数声明一起加进去：

```c
cmd_type_t get_cmd_type(const char *cmd_str);
int send_full(int fd, const void *buf, int len);
int recv_full(int fd, void *buf, int len);
void init_command_packet(command_packet_t *packet, cmd_type_t type, const char *data);
void init_file_packet(file_packet_t *packet, cmd_type_t type, const char *file_name, off_t file_size, off_t offset);
int send_command_packet(int fd, const command_packet_t *packet);
int recv_command_packet(int fd, command_packet_t *packet);
int send_file_packet(int fd, const file_packet_t *packet);
int recv_file_packet(int fd, file_packet_t *packet);
```

### 3.2 `src/common/protocol.c` 在旧代码中也是全新文件

旧代码中：**没有这个文件**

新代码中：**新增这个文件**

你需要新建 `old/src/common/protocol.c`。

这个文件主要提供 5 类能力：

1. 把命令字符串转成枚举值：`get_cmd_type`
2. 保证完整发送：`send_full`
3. 保证完整接收：`recv_full`
4. 初始化普通命令结构体：`init_command_packet`
5. 初始化文件传输结构体：`init_file_packet`

其中最重要的是这两段：

```c
int send_full(int fd, const void *buf, int len) {
    int total = 0;
    const char *p = (const char *)buf;
    while (total < len) {
        int ret = send(fd, p + total, len - total, MSG_NOSIGNAL);
        if (ret <= 0) {
            return -1;
        }
        total += ret;
    }
    return 0;
}
```

```c
int recv_full(int fd, void *buf, int len) {
    int total = 0;
    char *p = (char *)buf;
    while (total < len) {
        int ret = recv(fd, p + total, len - total, 0);
        if (ret <= 0) {
            return ret;
        }
        total += ret;
    }
    return total;
}
```

### 3.3 为什么一定先加协议层

因为后面这些文件都要用它：

1. `client_command_handle.c`
2. `server/handle.c`
3. `include/handle.h`

如果你不先加，后面会一直缺类型和函数声明。

## 四、修改头文件

### 第 2 步：改 `thread_pool.h`

这是线程池退出机制升级的前置条件。

### 4.1 旧代码

旧代码中的线程池结构体只有这些字段：

```c
typedef struct thread_pool{
    int num;
    pthread_t* thread_id_arr;
    queue_t queue;
    pthread_mutex_t lock;
    pthread_cond_t cond;
    int exitFlag;
}thread_pool_t;
```

### 4.2 新代码

新代码把线程池结构体改成了这样：

```c
typedef struct thread_pool thread_pool_t;

typedef struct worker_arg{
    thread_pool_t *pool;
    int index;
}worker_arg_t;

typedef struct thread_pool{
    int num;
    pthread_t* thread_id_arr;
    worker_arg_t *worker_arg_arr;
    queue_t queue;
    pthread_mutex_t lock;
    pthread_cond_t cond;
    int exitFlag;
    int *busy_fds;
}thread_pool_t;
```

还新增了这个函数声明：

```c
void destroy_thread_pool(thread_pool_t *pool);
```

### 4.3 为什么要这样改

旧代码的问题是：

1. 工作线程只知道线程池地址
2. 不知道“自己是第几个线程”
3. 线程池也不知道“哪个线程正在处理哪个 client_fd”

所以服务端退出时，没法对正在 `recv()` 的连接做 `shutdown`。

新代码通过这两个新增字段解决问题：

1. `worker_arg_arr`
   让每个线程知道自己的编号
2. `busy_fds`
   让主线程知道每个工作线程当前在处理哪个连接

### 第 3 步：改 `handle.h`

`handle.h` 的函数声明本身变化不大，主要是补充说明。

唯一需要注意的是：**后面的 `handle.c` 已经依赖 `protocol.h` 的思路了**，所以你改 `handle.h` 时，先保证函数声明和目标新代码一致即可。

如果你想省时间，这个文件可以最后再同步。

### 第 4 步：改 `old/src/client/client.c`

这个文件不是大改，主要有两件事：

1. 加日志初始化和关闭

```c
#include "client_socket.h"
```

### 5.3 手动修改要点

你只要改这几处：



## 六、重写客户端命令处理

### 第 5 步：重写 `old/src/client/client_command_handle.c`

这是客户端最核心的改造文件。

**建议：不要局部修修补补，直接按目标新代码整文件重写。**

原因是旧版和新版的通信协议完全不是一个思路。

### 6.1 旧代码的核心思路

旧代码一上来就是：

```c
int len = strlen(input);
send(sock_fd, &len, sizeof(int), 0);
send(sock_fd, input, len, 0);
```

然后 `gets` 和 `puts` 再额外混着收发 `off_t`、字符串响应。

### 6.2 新代码的核心思路

新版先把命令转成枚举，再统一发结构体：

```c
cmd_type_t cmd_type = get_cmd_type(cmd);
command_packet_t cmd_packet;
init_command_packet(&cmd_packet, cmd_type, arg);
send_command_packet(sock_fd, &cmd_packet);
```

### 6.3 这个文件必须完成的 6 个改造点

#### 改造点 1：增加协议头文件

旧代码：

```c
#include "client_command_handle.h"
```

新代码：

```c
#include "client_command_handle.h"
#include "protocol.h"
```

#### 改造点 2：增加两个辅助函数

新版新增：

1. `write_file_full`
2. `recv_server_reply`

关键新代码：

```c
static int write_file_full(int fd, const char *buf, int len)
```

```c
static void recv_server_reply(int sock_fd)
```

它们分别解决：

1. 本地写文件写不完整的问题
2. 普通文本响应统一接收的问题

#### 改造点 3：先校验参数，再发命令

旧代码的问题是：还没判断 `puts`/`gets` 参数是否合法，就已经把命令发出去了。

旧代码：

```c
int len = strlen(input);
send(sock_fd, &len, sizeof(int), 0);
send(sock_fd, input, len, 0);

if (strcmp(cmd, "gets") == 0) {
    if (strlen(arg) == 0) {
        printf("用法: gets <文件名>\n");
        return;
    }
}
```

新代码：

```c
if (strcmp(cmd, "gets") == 0 && strlen(arg) == 0) {
    printf("用法: gets <文件名>\n");
    return;
}

if (strcmp(cmd, "puts") == 0 && strlen(arg) == 0) {
    printf("用法: puts <文件名>\n");
    return;
}
```

#### 改造点 4：`gets` 变成“两次结构体握手”

旧代码 `gets`：

```c
off_t file_size = 0;
recv(sock_fd, &file_size, sizeof(off_t), MSG_WAITALL);
...
send(sock_fd, &local_size, sizeof(off_t), 0);
```

新代码 `gets`：

```c
file_packet_t server_file_packet;
recv_file_packet(sock_fd, &server_file_packet);

file_packet_t client_file_packet;
init_file_packet(&client_file_packet, CMD_TYPE_GETS, arg,
                 server_file_packet.file_size, request_offset);
send_file_packet(sock_fd, &client_file_packet);
```

也就是：

1. 服务端先发 `file_packet_t`
2. 客户端再回一个 `file_packet_t`

#### 改造点 5：`puts` 也变成“两次结构体握手”

旧代码 `puts`：

```c
send(sock_fd, &file_size, sizeof(off_t), 0);
recv(sock_fd, &server_offset, sizeof(off_t), MSG_WAITALL);
```

新代码 `puts`：

```c
command_packet_t cmd_packet;
init_command_packet(&cmd_packet, cmd_type, arg);
send_command_packet(sock_fd, &cmd_packet);

file_packet_t client_file_packet;
init_file_packet(&client_file_packet, CMD_TYPE_PUTS, arg, st.st_size, 0);
send_file_packet(sock_fd, &client_file_packet);

file_packet_t server_file_packet;
recv_file_packet(sock_fd, &server_file_packet);
```

#### 改造点 6：普通响应统一为 `command_packet_t`

旧代码：

```c
int resp_len = 0;
recv(sock_fd, &resp_len, sizeof(int), MSG_WAITALL);
recv(sock_fd, response, resp_len, MSG_WAITALL);
```

新代码：

```c
recv_server_reply(sock_fd);
```

底层实现改为：

```c
command_packet_t reply_packet;
recv_command_packet(sock_fd, &reply_packet);
printf("%s\n", reply_packet.data);
```

### 6.4 手工修改建议

这个文件建议你按下面方式改：

1. 先保留旧文件备份
2. 直接对照目标新文件，把整个文件重写
3. 重写顺序按函数来抄：
   1. include 区
   2. `write_file_full`
   3. `recv_server_reply`
   4. `process_command`

### 6.5 为什么我建议整文件重写

因为这已经不是“小补丁”级别改动了，而是：

1. 通信协议变了
2. `gets` 流程变了
3. `puts` 流程变了
4. 普通命令响应格式也变了

如果你坚持局部 patch，反而更容易改乱。

## 八、改服务端入口

### 第 7 步：改 `old/src/server/server.c`

这个文件的主要变化有 4 类：

1. 增加日志初始化
2. 增加 `#include <sys/socket.h>`
3. 增强线程池退出流程
4. 退出时销毁线程池、关闭日志

### 8.1 旧代码关键片段

```c
signal(SIGPIPE, SIG_IGN);
...
if(fd == pipe_fd[0]){
    ...
    pthread_mutex_lock(&pool.lock);
    pool.exitFlag = 1;
    pthread_cond_broadcast(&pool.cond);
    pthread_mutex_unlock(&pool.lock);

    for(int i = 0; i < pool.num; i++){
        pthread_join(pool.thread_id_arr[i], NULL);
    }
    pthread_exit((void*)NULL);
}
```

### 8.2 新代码关键片段

```c
init_log("INFO", NULL);
signal(SIGPIPE, SIG_IGN);
```

并且退出部分改成：

```c
pthread_mutex_lock(&pool.lock);
pool.exitFlag = 1;

while(pool.queue.size > 0){
    int client_fd = deQueue(&pool.queue);
    if(client_fd != -1){
        shutdown(client_fd, SHUT_RDWR);
        close(client_fd);
    }
}

for(int i = 0; i < pool.num; i++){
    if(pool.busy_fds[i] != -1){
        shutdown(pool.busy_fds[i], SHUT_RDWR);
    }
}

pthread_cond_broadcast(&pool.cond);
pthread_mutex_unlock(&pool.lock);

close(listen_fd);

for(int i = 0; i < pool.num; i++){
    pthread_join(pool.thread_id_arr[i], NULL);
}

destroy_thread_pool(&pool);
close(pipe_fd[0]);
close(pipe_fd[1]);
close(epfd);
close_log();
return 0;
```

### 8.3 为什么退出逻辑要改

旧代码的问题是：

1. 只设置 `exitFlag`
2. 只唤醒睡眠中的线程

但是 **正在 `recv()` 的线程不会自动退出**。

新代码通过两件事解决：

1. 主动关掉队列里还没处理的连接
2. 主动 `shutdown` 正在处理的连接

这样阻塞在 `recv()` 的线程更容易返回。

### 8.4 手改建议

这个文件不是整文件重写型，但你最好按下面顺序改：

1. 加 `#include <sys/socket.h>`
2. 在 `main` 开头加 `init_log("INFO", NULL);`
3. 把 `pipe_fd[0]` 分支里的退出逻辑整体替换成新逻辑
4. 在函数结尾补 `close_log();`

## 九、改线程池与工作线程

### 第 8 步：改 `old/src/server/thread_pool.c`

这个文件要配合 `thread_pool.h` 一起改。

### 9.1 旧代码

旧代码创建线程时直接把 `pool` 传给线程：

```c
pool->thread_id_arr=(pthread_t*)malloc(num*sizeof(pthread_t));

for(int idx=0;idx<num;++idx){
    int ret=pthread_create(&pool->thread_id_arr[idx],NULL,thread_func,(void*) pool);
    ERROR_CHECK(ret,-1,"pthread_create");
}
```

### 9.2 新代码

新代码先多分配两块内存：

```c
pool->worker_arg_arr=(worker_arg_t*)malloc(num*sizeof(worker_arg_t));
pool->busy_fds=(int*)malloc(num*sizeof(int));
```

然后线程创建方式改成：

```c
pool->worker_arg_arr[idx].pool=pool;
pool->worker_arg_arr[idx].index=idx;
int ret=pthread_create(&pool->thread_id_arr[idx],NULL,thread_func,
                       (void*)&pool->worker_arg_arr[idx]);
THREAD_ERROR_CHECK(ret,"pthread_create");
```

并新增：

```c
void destroy_thread_pool(thread_pool_t *pool)
```

### 9.3 这里有一个非常关键的小改动

旧代码错误地用了：

```c
ERROR_CHECK(ret,-1,"pthread_create");
```

新版改成：

```c
THREAD_ERROR_CHECK(ret,"pthread_create");
```

因为 `pthread_create` 失败时返回的是 **正的错误码**，不是 `-1`。

### 第 9 步：改 `old/src/server/worker.c`

### 9.4 旧代码

旧代码线程函数签名里，把参数直接当 `thread_pool_t *` 用：

```c
void* thread_func(void *arg) {
    thread_pool_t *pool = (thread_pool_t *)arg;
```

### 9.5 新代码

新版先转成 `worker_arg_t *`：

```c
worker_arg_t *worker_arg = (worker_arg_t *)arg;
thread_pool_t *pool = worker_arg->pool;
int worker_index = worker_arg->index;
```

并且在取到任务后增加：

```c
pool->busy_fds[worker_index] = client_fd;
```

在处理完成后增加：

```c
shutdown(client_fd, SHUT_RDWR);
close(client_fd);

pthread_mutex_lock(&pool->lock);
pool->busy_fds[worker_index] = -1;
pthread_mutex_unlock(&pool->lock);
```

### 9.6 手改建议

这两个文件最好一起改：

1. 先改 `thread_pool.h`
2. 再改 `thread_pool.c`
3. 最后改 `worker.c`

## 十、重写服务端命令处理

### 第 10 步：重写 `old/src/server/handle.c`

这是整个迁移里最大的一个文件。

**建议：直接按目标新代码整文件重写。**

### 10.1 为什么这个文件必须重写

因为它同时发生了 5 类变化：

1. 普通响应协议变了
2. 命令接收协议变了
3. 命令分发方式变了
4. `gets` 和 `puts` 的文件传输协议变了
5. 路径安全和字符串拼接方式变了

### 10.2 旧代码的 3 个核心问题

#### 问题 1：命令协议是“长度 + 字符串”

旧代码：

```c
int cmd_len=0;
recv(listen_fd,&cmd_len,sizeof(int),MSG_WAITALL);
recv(listen_fd,cmd_buf,cmd_len,MSG_WAITALL);
sscanf(cmd_buf,"%s %s",cmd,arg);
```

新代码：

```c
command_packet_t cmd_packet;
recv_command_packet(listen_fd, &cmd_packet);
cmd_type_t cmd_type = get_packet_cmd_type(&cmd_packet);
```

#### 问题 2：命令分发靠一长串 `strcmp`

旧代码：

```c
if(strcmp(cmd,"pwd")==0){
    handle_pwd(...);
}else if(strcmp(cmd,"cd")==0){
    ...
}
```

新代码：

```c
switch (cmd_type) {
case CMD_TYPE_PWD:
    handle_pwd(...);
    break;
case CMD_TYPE_CD:
    handle_cd(...);
    break;
...
}
```

#### 问题 3：`send_msg` 还是“长度 + 字符串”

旧代码：

```c
int len = strlen(msg);
send(listen_fd, &len, sizeof(int), MSG_NOSIGNAL);
send(listen_fd, msg, len, MSG_NOSIGNAL);
```

新代码：

```c
command_packet_t reply_packet;
init_command_packet(&reply_packet, CMD_TYPE_REPLY, msg);
send_command_packet(listen_fd, &reply_packet);
```

### 10.3 `gets` 必须怎么改

旧代码 `gets` 的协议是：

1. 服务端直接发 `off_t file_total_size`
2. 客户端回一个 `off_t offset`
3. 服务端开始 `sendfile`

新代码变成：

1. 服务端先发 `file_packet_t`
2. 客户端再回 `file_packet_t`
3. 然后服务端再 `sendfile`

旧代码片段：

```c
off_t file_total_size = st.st_size;
send(listen_fd, &file_total_size, sizeof(off_t), 0);

off_t offset = 0;
recv(listen_fd, &offset, sizeof(off_t), MSG_WAITALL);
```

新代码片段：

```c
file_packet_t server_file_packet;
init_file_packet(&server_file_packet, CMD_TYPE_GETS, arg, st.st_size, 0);
send_file_packet(listen_fd, &server_file_packet);

file_packet_t client_file_packet;
recv_file_packet(listen_fd, &client_file_packet);
off_t offset = client_file_packet.offset;
```

### 10.4 `puts` 必须怎么改

旧代码 `puts` 的协议是：

1. 客户端发 `off_t file_len`
2. 服务端回 `off_t local_size`
3. 客户端直接发文件内容

新代码变成：

1. 客户端先发普通命令结构体
2. 再发文件结构体
3. 服务端回文件结构体（带 offset）
4. 客户端从 offset 继续发

旧代码片段：

```c
off_t file_len = 0;
recv(listen_fd, &file_len, sizeof(off_t), MSG_WAITALL);
...
send(listen_fd, &local_size, sizeof(off_t), 0);
```

新代码片段：

```c
file_packet_t client_file_packet;
recv_file_packet(listen_fd, &client_file_packet);

file_packet_t server_file_packet;
init_file_packet(&server_file_packet,
                 CMD_TYPE_PUTS,
                 arg,
                 client_file_packet.file_size,
                 local_size);
send_file_packet(listen_fd, &server_file_packet);
```

### 10.5 路径处理必须怎么改

旧代码路径拼接很直接：

```c
void get_real_path(char *res, const char *path, const char *arg) {
    if (strcmp(path, "/") == 0) {
        sprintf(res, "%s/%s", SERVER_BASE_DIR, arg);
    } else {
        sprintf(res, "%s%s/%s", SERVER_BASE_DIR, path, arg);
    }
}
```

新代码增加了：

1. `check_arg_path`
2. `get_current_real_path`
3. `get_real_path`
4. `update_current_path`

核心区别是：

1. 拒绝 `..`
2. 改用 `snprintf`
3. 对路径长度做检查

## 十七、你手改时最容易犯的 8 个错误

1. 只新增了 `protocol.h`，忘了新增 `protocol.c`
2. `client_command_handle.c` 还保留旧版 `send(len) + send(input)` 逻辑
3. `server/handle.c` 还保留旧版 `recv(cmd_len) + recv(cmd_buf)` 逻辑
4. `puts` 仍然只收发 `off_t`，没有改成 `file_packet_t`
5. `gets` 文件不存在时仍然发字符串，而不是 `file_packet_t(file_size=-1)`
6. `pthread_create` 仍然用 `ERROR_CHECK(ret,-1,...)`
7. 线程池结构体里忘记加 `busy_fds`
8. 服务端退出时忘了 `shutdown(pool.busy_fds[i], SHUT_RDWR)`

如果这 8 个坑都避开了，迁移基本就稳了。

## 十八、最后一句建议

你的目标不是“把 old 代码勉强改到能编过”，而是：

1. 旧协议彻底换成新协议
2. 线程池退出逻辑跟上新版本
3. 上传下载流程真正和新代码一致

所以在真正动手时，最省力的方法不是死抠所有小 diff，而是：

1. **协议层直接新增**
2. **两大核心文件直接重写**
3. **线程池和服务端主流程按模块同步**

这样最接近当前根目录中的目标新代码，也最不容易改乱。
